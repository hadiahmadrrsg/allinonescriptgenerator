🚀 Quick Start Guide

Generate a professional CS script in 4 simple steps:

1
Select Dispute Type & Result

Choose Buyer Request or Seller Dispute, then pick the request type (Return & Refund / Refund Only / With Buyer / Without Buyer), and finally select the decision result (Accept, Reject, Request Evidence, etc.)

2
Fill in Case Details

Enter the Buyer Username, Seller Username, Order SN, Product Name, and select the Return Reason. For Accept Request, enter the Refund Amount. You can also use the DMS Auto-Fill to populate these automatically.

3
Click Generate Script

The tool assembles a randomized professional script from 200+ templates. Each paragraph (Empathy, Result Sharing, CSAT Pitching, Magic Word) is independently randomized.

4
Review, Edit & Copy

Review the output. Use Reshuffle to get a different version of any paragraph, Edit to modify text, + Add to insert custom paragraphs, or Delete to remove sections. Then click Copy All to paste into DMS chat.

All in One Script Generator

How the tool works internally — so anyone can understand and fix it.

🗺️ Big Picture

The generator is a single HTML file with no backend. Everything — scripts, logic, admin panel, evidence collection, snippet library — lives inside one file. Here's the end-to-end flow:

Agent fills form
→
Click Generate
→
Look up SCRIPT_ASSEMBLY
→
Pick random scripts from RAW_DB
→
Inject live values
→
Show output
📋 Form inputs
  • User Type (Buyer / Seller)
  • Request Type
  • Decision / Result
  • Username, Order SN, Product
  • Refund Amount / SLA Wallet
  • Return Reason / Dispute Reason
  • Root Cause / Request Evidence / Next Steps
🔧 Key JS variables
  • state — all current form values
  • RAW_DB — all 350 script entries
  • SCRIPT_ASSEMBLY — which sections each flow uses
  • BUYER_REASONS / SELLER_REASONS — dropdown options
  • db — live database (editable via Admin)

🗄️ Data Structure

Every script in the tool is stored as a 6-column array inside RAW_DB. Think of it like a spreadsheet row.

// RAW_DB entry format:
[
  "Type Return",      // col 0: e.g. "Return & Refund, Refund Only" or "Seller Dispute"
  "Script Type",      // col 1: "Type A (With Emoji)"
  "User",             // col 2: "Buyer" or "Seller"
  "Result",           // col 3: e.g. "Accept Request, Close Request"
  "Section/Item",     // col 4: e.g. "Empathy + Issue Statement"
  "Script Text"       // col 5: the actual script content
]

Example:

[
  "Return & Refund, Refund Only",
  "Type A (With Emoji)",
  "Buyer",
  "Accept Request, Close Request",
  "Empathy + Issue Statement",
  "Hi {username}, thanks for reaching out.\nWe're sorry to hear about your concern with order {order_sn}..."
]
⚠️ The Result column can contain multiple values — e.g. "Accept Request, Close Request" means this script is used for BOTH flows. The matching is done with an includes() check, not exact match.

🔀 Flows & Assembly

SCRIPT_ASSEMBLY defines which sections appear for each flow, and which scripts to pick for each section.

// Format: "User|Result" → array of sections
"Buyer|Accept Request": [
  { qR: "Accept Request, Close Request", item: "Empathy + Issue Statement" },
  { qR: "Accept Request",              item: "Result Sharing + Root Cause" },
  { qR: "Accept Request, Close Request", item: "CSAT Pitching" },
  { qR: "Accept Request, Close Request", item: "Magic Word / Closing" }
]

The qR value is used to filter RAW_DB — it must exactly match the Result column of the script entry.

⚡ Script Generation

When the agent clicks Generate Script, the function generateScript() runs:

  1. Reads state to know User, Type, Result, and all field values
  2. Looks up SCRIPT_ASSEMBLY["User|Result"] to get the list of sections
  3. For each section, calls getPool(typeReturn, typeScript, user, qR, item) which filters db (live database)
  4. Picks a random script from each pool using rand(pool)
  5. Replaces {placeholders} via the fill() function
  6. Injects live field values (refund amount, SLA wallet, root cause etc.) directly into the text
  7. Saves to state.generatedSections and calls renderOutput()
// Simplified logic inside generateScript()
const steps = SCRIPT_ASSEMBLY[`${state.user}|${state.result}`];
const sections = steps.map(step => {
  const pool = getPool(typeReturn, typeScript, user, step.qR, step.item);
  const script = fill(rand(pool), vars);
  return { item: step.item, text: script };
});
Where to look if a section is blank or missing: Check that RAW_DB has entries where col 3 (Result) exactly matches the qR value in SCRIPT_ASSEMBLY, col 2 = correct User, and col 0 includes the Request Type.

After scripts are assembled, live field values are stitched in directly inside generateScript():

Data Structure — EVIDENCE_DB

All evidence data lives in a single nested JSON object called EVIDENCE_DB with the following hierarchy:

// EVIDENCE_DB structure
EVIDENCE_DB = {
  template: "Base template with [Evidence Required] placeholder",
  seller:    { // Seller Forward Leg
    "Return Reason": {
      "Risk Level": {
        "Evidence Type": "Full script text..."
      }
    }
  },
  returnLeg: { // Seller Return Leg — same structure },
  buyer:     { // Buyer Evidence — same structure },
  bullets:   { // Short bullet-point evidence requirements
    seller:    { /* mirrors seller structure */ },
    returnLeg: { /* mirrors returnLeg structure */ },
    buyer:     { /* mirrors buyer structure */ }
  }
}

Three Categories

🛒 Buyer — Evidence

Evidence required FROM the buyer. Covers: Incomplete Order, Wrong Product, Physical Damaged, Functional Damaged, Expired Product, Empty Parcel, Return in Original Condition (COM), Counterfeit, Virtual Goods, Escalation to 3PL.

🏪 Seller — Forward Leg

Evidence required FROM the seller about the outbound shipment. Same return reasons but with different risk tiers (Low/Medium, High, Non-integrated) and evidence types (3PL Related, Weight Verification, etc.).

🔄 Seller — Return Leg

Evidence required FROM the seller about what they received back. Covers: Incomplete / Wrong / Damaged returns at Low/Medium and High Risk tiers (MAIN, Weight Verification, Missing Part, Faulty).

Risk Levels by Category

CategoryRisk Levels
BuyerLow / Medium Risk Abusive Potential Fraud
Seller ForwardLow / Medium Risk High Risk Non-integrated
Seller ReturnLow / Medium Risk High Risk

Full Scripts vs Bullets

EVIDENCE_DB.seller, .buyer, .returnLeg contain the full scripts (complete template with header, evidence list, and 48-hour reminder). EVIDENCE_DB.bullets contains short bullet-point summaries of just the evidence requirements — these are shown in the Admin panel and used by the Evidence Collection tab's Copy button.

CSV Export/Import Format

The Admin panel supports exporting and importing evidence via CSV with 6 columns:

Category | Return Reason | Risk Level | Evidence Type | Evidence Bullets | Full Script

🐛 Common Errors & Where to Fix

Error  Section is blank / script missing

A section generates nothing or shows placeholder text.

Fix: Open Admin → filter by the broken flow → check if scripts exist for that Section. The qR in SCRIPT_ASSEMBLY must exactly match the Result column in RAW_DB.

Error  RAW_DB / BUYER_REASONS already declared

Page doesn't load, console shows duplicate declaration error.

Fix: The HTML file has been duplicated internally. Search for const RAW_DB — there should only be one. Delete the first occurrence (the smaller one).

Issue  Bookmarklet doesn't run on DMS

Clicking the bookmark does nothing, or prompt doesn't appear.

Fix: Regenerate the bookmarklet in Admin. The stored bookmark may have been corrupted by the browser during drag-and-drop. Use "Copy Code" → create bookmark manually → paste code as URL.

Issue  DMS Auto-Fill fills fields but Decision/Result is not selected

This is by design — the agent must select the result manually.

Not a bug. state.result = '' is set intentionally in autoFillFromDMS() so the agent picks the correct decision themselves.

Issue  Button doesn't highlight when clicked (Seller)

Seller result buttons show no active colour.

Fix: The setResult() function must compare using button.dataset.value, not button.textContent — because Seller labels are renamed (e.g. "Accept Request" → "Accept Request") but the internal value stays the same.

Error  url is not defined (console)

Page loads with a console error about url.

Fix: The bmCode variable uses a template literal with ${url} which JavaScript evaluates at parse time. The bookmarklet template must use a plain string placeholder like __SCRIPT_GENERATOR_URL__ instead.

✏️ How to Edit Scripts

Method 1 — Admin Panel (recommended)

  1. Open the generator → click ⚙️ Admin
  2. Use the filters to find the script you want to edit
  3. Click ✏️ Edit → modify the text → click Save
  4. Changes take effect immediately in the generator
⚠️ Changes made in Admin are lost when the HTML file is replaced. Export a CSV backup first, or use Import Sheet to re-apply changes after an update.

Method 2 — Edit RAW_DB directly in the HTML

  1. Open the HTML file in a code editor (VS Code, Notepad++)
  2. Search for const RAW_DB = [
  3. Each script entry is a 6-element array — find the one to edit using Ctrl+F with part of the script text
  4. Edit col index [5] (the script text)
  5. Save and reload the page

Adding a new flow or section

  1. Add the script entries to RAW_DB with the correct 6 columns
  2. If it's a new flow (new User+Result combination), add it to SCRIPT_ASSEMBLY with the sections and their qR values
  3. If the new result is a Seller flow with a renamed label, update resultLabel in renderForm()
  4. Add the result to the results array in renderForm()
✅ Quick test: After making changes, open the browser console (F12) and check for errors. Then try generating a script for every flow that was changed to confirm scripts appear correctly.
